## Сценарии — архитектура БД

```text
oca_telegram_bot_scenarios
  -> базовая карточка сценария

oca_telegram_bot_scenario_bots
  -> подключение одного или нескольких каналов к сценарию

oca_telegram_bot_scenario_blocks
  -> будущие блоки на рабочем поле

oca_telegram_bot_scenario_links
  -> будущие связи между блоками

oca_telegram_bot_scenario_block_cards
  -> будущие карточки внутри блока «Сообщение»

oca_telegram_bot_scenario_card_buttons
  -> будущие кнопки карточек

oca_telegram_bot_scenario_deeplinks
  -> один основной диплинк на блок

oca_telegram_bot_subscriber_scenarios
  -> прохождение сценария конкретным подписчиком через конкретный bot_id

oca_telegram_bot_scenario_events
  -> журнал событий сценария
```

## Важные правила сценариев

```text
1. Сценарий — глобальная сущность.
2. К сценарию можно подключить один или несколько каналов.
3. Конкретное прохождение подписчика всегда идёт через конкретный bot_id.
4. Один telegram_user_id может проходить один сценарий в разных каналах.
5. В одном канале один человек не должен одновременно проходить один и тот же сценарий дважды.
6. Диплинк создаётся на блок, основной диплинк один.
7. Диплинк формирует Telegram-ссылку вида https://t.me/{bot_username}?start=dl-...
8. По диплинку человек стартует сценарий с конкретного блока.
```

## Следующая зона разработки

```text
Шаг 3 из 10 — редактор блоков отдельным файлом.
```

Следующий патч не должен менять основную страницу списка сценариев. Редактор блоков нужно вынести отдельно:

```text
admin_app/modules/telegram_bots/pages/scenario_editor.php
```

Вход:

```text
/admin.php?tab=telegram_bots&page=scenarios&scenario_id=...&editor=1
```

или отдельный route:

```text
/admin.php?tab=telegram_bots&page=scenario_editor&scenario_id=...
```

Главное правило: список сценариев больше не ломать.
